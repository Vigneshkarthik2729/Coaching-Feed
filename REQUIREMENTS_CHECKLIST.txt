EVENT FLOW:
1. User submits feed on /admin
2. POST request to /feed endpoint
3. Server creates feed in MongoDB
4. Server clears Redis cache with: redisClient.del("feeds")
5. Server broadcasts with: io.emit("newFeed", feed)
6. Client receives event via: socket.on("newFeed", ...)
7. Client updates state and highlights new feed
8. All connected clients see update instantly

IMPLEMENTATION:
Server:
  - io.emit("newFeed", feed) - Broadcasts to all clients
  
Client:
  - socket.on("newFeed", (newFeed) => {...}) - Listen for events
  - Cleanup with socket.off("newFeed") on unmount

REDIS USAGE SUMMARY
================================================================================

WHERE:
- Config: server/config/redis.js
- Routes: server/routes/feedRoutes.js (GET and POST)

WHY:
1. Performance: 10-20x faster reads with caching
2. Scalability: Reduces database queries
3. Consistency: Cache invalidation ensures fresh data
4. Reliability: Error handling for Redis failures

CACHE STRATEGY:
- Key: "feeds"
- Value: JSON stringified array of feed objects
- Invalidation: Deleted on feed creation
- TTL: None (cleared on mutations)

DATA FLOW:
GET /feed:
  1. Check Redis cache: redisClient.get("feeds")
  2. If exists: Parse and return (FAST: ~5-10ms)
  3. If not: Query MongoDB (SLOW: ~100-200ms)
  4. Store in Redis: redisClient.set("feeds", JSON.stringify(feeds))
  5. Return to client

POST /feed:
  1. Save to MongoDB: Feed.create({...})
  2. Clear cache: redisClient.del("feeds")
  3. Broadcast event: io.emit("newFeed", feed)
  4. Return created feed

PERFORMANCE IMPACT:
- First GET: 100-200ms (DB + cache set)
- Next GETs: 5-10ms (cache hit)
- Improvement: 10-20x faster
